Novice to Ninja Chapter 10 - Testing and Debugging

Errors

System error - there’s a problem with the system or external devices with which the program is interacting.

Programmer error - the program contains incorrect syntax or faulty logic; it could even be as simple as a typo.

User error - the user has entered data incorrectly, which the program is unable to handle.

Exceptions

An exception is an error that produces a return value that can be used by the program to deal with the error.

Stack traces help you find where an erorr orignates from. Remember the example:
function three(){ unicorn(); }
function two(){ three(); }
function one(){ two(); }
one();

output:
index.html:13 Uncaught ReferenceError: unicorn is not defined
at three (index.html:13)
at two (index.html:17)
at one (index.html:21)
at index.html:24`

Warnings

A warning can occur if there’s an error in the code that isn't enough to cause the program to crash. This means the program will continue to run after a warning.

Warnings may be problematic because the program continues to run. This can be avoided with proper user testing.

Why is Testing and Debugging Important?

Failing gracefully rather than silently is the goal. When you fail silently things can be missed and overlooked than cause issues down the road.

Strict mode can be used in order to increase the ability to catch errors. When you aren't in strict mode you are in "sloppy mode" because sloppy proctices can sneak through

To use strict mode place 'use strict'; at the top line of your js file. You can also place 'use strict'; inside a function.

The suggested way to do this is:
(function() {
'use strict';

// All your code would go inside this function

}());

Linting tools are a great way to adhere to certain conventions. There are online or add-on linters that can be used.

To find if browser supports a certain feature you can use an if statement a library like Modernizir or broswer sniffing. This is refered to as "Feature Detection"

Debugging is the process of finding out where bugs occur in the code and then dealing with them.

This is usually done with manual coding or using developer tools to set breakpoints. Alerts can return the value of a certain variable. The console can also be used to do the same thing. Most modern browsers have debugging tools that can be used.

The 'debugger' keyword will create a breakpoint in your code that will pause the execution of the code and allow you to see where the program is currently up to. You can also hover over any variables to see what value they hold at that point. You should be sure to remove references to the debugger command before deploying.

    Error Objects:

    • Properties for these in most browsers are "name", "message" and "stack".

You can throw your own exception using the 'throw' statement. You can also manually code a try and catch blocks to be a sort of error handler. The try holds a piece of code we think may fail. The catch block can pass then pass the error if that happens.

Test, Test and Test Some More!

By testing with system, programmer and user errors in mind you can avoid headaches in the future.

    Test-driven development (TDD) is the process of writing tests before any actual code. The process for this is as follows:
  1. Write tests (that initially fail)
  2. Write code to pass the tests
  3. Refactor the code
  4. Test refactored code
  5. Write more tests for new features

You can make your own tests but this can be a hard task. To help with this Testing Frameworks have been developed.
